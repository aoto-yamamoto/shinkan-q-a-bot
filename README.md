# 新歓Q&A自動応答システム

サークル新歓の問い合わせ対応を自動化する、LINE風チャットUIのQ&A Botです。
新入生からの質問をLLM（Claude API）で意図分類し、FAQに該当すれば即時回答、
該当しなければ担当者へエスカレーションします。

## 背景・目的

新歓統括を担当した際、「新入生からの質問に回答しきれない」「回答を届ける手段がない」
という課題があり、その解決のために要件定義から設計・実装・運用まで一人で行いました。

- 実際の利用者（新入生）と発注者（サークル）が存在する実運用プロジェクト
- 公開期間：2027年2月〜4月（新歓当日まで）、想定利用 1〜5件/時

## 特徴

- **誤案内ゼロを優先した設計**：LLMの役割を「質問の意図分類」に限定し、
  回答は事前承認済みの定型文のみを返す（LLMに自由生成させない）
- **段階的な個人情報収集**：メールアドレスは「担当者からの個別回答」を
  希望した利用者からのみ、必要になった時点で取得
- **プロンプトの事前検証**：実装前に27問の検証リスト（言い換え・境界ケース・
  プロンプトインジェクションを含む）で判定精度を測定し、誤案内0件を確認
- **運用を考えた作り**：スクリプトプロパティによるメンテナンスモード、
  セッション単位のレート制限、APIコスト上限の設計

## システム構成

```
新入生（スマホブラウザ）
   │ 質問
   ▼
GAS Webアプリ（フロント + バック）
   ├─→ Claude API（意図分類のみ・Haiku系軽量モデル）
   └─→ Google スプレッドシート（全質問を記録）
            ▲
        担当者が1日2回確認 → 未対応の質問にメール/公式LINEで回答
```

- フロント/バック: Google Apps Script（Webアプリ）
- 意図分類: Claude API（Messages API）
- データ記録: Google スプレッドシート
- 選定理由: サーバレス・運用費ゼロ・スプレッドシート標準連携（詳細は docs/基本設計書 参照）

## 工夫した点・学んだこと


- 検証で「FAQに近いが答えられない質問」（開催場所など）の誤案内が最も危険と気づき、
  判定ルールに境界条件を明記した
- １度実装した後に同一ブラウザに対する返答回数を変更した。実装試験でのシステムの変更の重要性
　を身をもって体験できた
  具体的には実装後のテストで、FAQチップ数（6個）に対してレート制限5回では不足すると気づき、上限を10回に変更した。
  テストを通じた仕様の見直しの重要性を体験した
- フロントの入力検証はサーバ側でも二重に行う理由を実装を通じて理解した
- 運用開始後に気づいたこと（本番運用後に追記します）
